# 授信审批意见生成助手（内部草稿 + 条款清单）

本技能用于快速产出“审批会/会签可用”的**审批意见草稿**，重点不是替代审批，而是：
- 把“申请要素—风险点—缓释—条款条件”整理成结构化文本
- 明确哪些是已确认事实，哪些是待核验事项
- 把“有条件同意”写成可执行的**提款前条件**与**贷后监测触发条款**

输出适合：审查意见初稿、审批会准备材料、条款清单整理、会签沟通底稿。

## 适用范围
- 授信审批意见草稿（同意/有条件同意/暂缓/否决建议）
- 条款与条件梳理：提款前条件、贷后监测、触发条款、资金闭环
- 审批会沟通材料：风险点与争议点清单

## 何时使用
- 你需要在短时间内把材料整理成审批意见初稿（结构统一、便于会签修改）
- 需要把风险点落到可执行条款（而不是停留在“提示风险”）
- 需要明确“暂缓/否决”的依据与缺口清单（便于补数与复核）

## 何时不要使用
- 用户要求你输出“最终批复/最终审批结论”时（本 skill 只产出草稿）
- 用户要求伪造财务、授信、尽调、合规材料或隐瞒重大风险时
- 关键信息缺口过大且无法补齐时（只能输出框架 + 缺口）

## 默认工作流
1. **申请要素梳理**：产品、金额、期限、用途、还款来源、担保押品、提款条件。
2. **风险点清单**：必须可核验（证据/口径/来源）；区分“重大/一般/待确认”。
3. **缓释措施**：必须可执行（条款化、流程化、监测化），避免空泛“加强管理”。
4. **意见建议**：同意/有条件同意/暂缓/否决的逻辑链条与依据。
5. **条款清单**：提款前条件、贷后监测指标、触发条款、信息披露义务。
6. **边界声明**：草稿用途、待核验事项、升级会签路径。

## 重点分析框架
- 信息来源是否清楚，时间线是否清晰
- 摘要是否保留关键风险、决策点和责任分工
- 是否明确哪些事项仍未解决
- 输出是否适合目标受众快速阅读

## 输入要求
最低必需（详见 `references/input-schema.md`）：
- `company.name`
- `application.product_type` / `application.amount_mn` / `application.term` / `application.purpose`
- `risk_points[]`（若缺失，则只能输出框架）

建议提供：
- `mitigants[]`：风险缓释措施（条款/流程/监测）
- `proposed_terms[]`：条款建议或已初步拟定条款
- `gaps[]`：明确缺口，脚本会纳入意见草稿

## 输出要求
- 建议结论：同意/有条件同意/暂缓/否决（草稿建议）
- 审批意见 Markdown 初稿（可会签改写）
- 提款前条件建议（conditions_precedent）
- 贷后监测与触发条款建议（post_approval_monitoring）
- 主要风险点、缓释措施、条款建议（输入 + 结构化整理）
- 关键信息缺口（gaps）

脚本输出字段见 `references/output-schema.md`。

## 风险与边界
- 不得把初步分析包装成正式授信批复或最终审批结论
- 不得编造财务、行业或外部查询结果
- 不得遗漏重大风险、分歧点或未决事项
- 不清楚的内容应标记为待确认，不得擅自补写

## 信息不足时的处理
- 先列出已经掌握的事实，再明确缺失的关键字段，不要直接沉默
- 无法形成强结论时，优先输出框架、待补资料和优先核验事项
- 对依赖外部数据、制度文本或人工核实的部分，要单独标注[待补充/待确认]

## 配套脚本（可重复执行）

```bash
python scripts/run_skill.py --input assets/example-input.json --format markdown
python scripts/run_skill.py --input assets/example-input.json --format json
```

- `scripts/run_skill.py`：命令行入口
- `scripts/approval_opinion_drafter.py`：t144 场景封装（调用共享引擎）

## 交付标准
- 风险点必须可核验，缓释必须可执行（条款化/流程化/监测化）
- 草稿必须含：项目要点、风险点、缓释与条件、条款建议、边界声明
- 能直接在审批会/会签中作为“底稿”快速改写
