# 授信问题清单生成助手（补件/催办/复查闭环）

本技能把“审查提问/补件要求/风险关注点”整理成**可推进的台账**：
- 把问题按严重程度分层：高（硬性阻断）/中（关键证据缺口）/低（一般补充）
- 每条问题必须包含：**问题本体、证据要求、影响、责任人、截止时间**
- 输出要能直接用于：催办、复查、会签升级、记录归档

## 适用范围
- 授信准入/续作：审查问题清单与补件清单
- 贷后核验：预警触发后的问题分解与整改跟踪
- 中台运营：标准化催办台账，避免“口头问、口头回”

## 何时使用
- 需要把一堆审查意见整理成“可催办”的台账
- 需要明确：哪些是硬性阻断项、哪些是补强证据后可推进项
- 需要为每条问题设定责任归口、截止时间与复查节点

## 何时不要使用
- 用户要求弱化/删除关键风险、把待确认写成已确认时
- 需要替代审批结论时（本 skill 只输出问题与证据要求，不下最终批复）

## 默认工作流
1. 收集问题来源：审查意见、访谈纪要、台账、外部查询、贷后检查等。
2. 结构化每条问题：问题描述、证据、影响、责任人、截止时间。
3. 分级：硬性阻断项（高）优先闭环；关键证据缺口（中）设定最小证据；一般补充（低）合并批量处理。
4. 输出台账：按严重程度排序，形成催办节奏与升级路径。
5. 复查闭环：收到材料后逐条更新“证据是否满足/是否仍有缺口/是否需要升级”。

## 重点分析框架
- 信息来源是否清楚，时间线是否清晰
- 摘要是否保留关键风险、决策点和责任分工
- 是否明确哪些事项仍未解决
- 输出是否适合目标受众快速阅读

## 输入要求
最低必需（详见 `references/input-schema.md`）：
- `company.name`
- `issue_context.stage`（准入/续作/贷后/重组等）
- `issues[]`（每条至少包含 `issue` 与 `severity`）

强烈建议：
- `issues[].required_materials`：最小证据清单
- `issues[].owner` 与 `issues[].due_date`：便于催办闭环

## 输出要求
- 硬性阻断项（hard_stoppers）
- 问题清单台账（issues：含严重度、责任人、截止时间）
- 补件/证据建议（supplement_suggestions）
- 核验/访谈问题（questions）
- 催办与推进动作（actions）
- 关键信息缺口（gaps）

脚本输出字段见 `references/output-schema.md`。

## 风险与边界
- 不得把初步分析包装成正式授信批复或最终审批结论
- 不得编造财务、行业或外部查询结果
- 不得遗漏重大风险、分歧点或未决事项
- 不清楚的内容应标记为待确认，不得擅自补写

## 信息不足时的处理
- 先列出已经掌握的事实，再明确缺失的关键字段，不要直接沉默
- 无法形成强结论时，优先输出框架、待补资料和优先核验事项
- 对依赖外部数据、制度文本或人工核实的部分，要单独标注[待补充/待确认]

## 配套脚本（可重复执行）

```bash
python scripts/run_skill.py --input assets/example-input.json --format markdown
python scripts/run_skill.py --input assets/example-input.json --format json
```

- `scripts/run_skill.py`：命令行入口
- `scripts/credit_issue_list_builder.py`：t143 场景封装（调用共享引擎）

## 交付标准
- 每条问题都可执行：责任到人 + 截止时间 + 证据交付物
- 能在 1 页内看清楚“高优先级/硬阻断”是什么，以及下一步怎么推进
- 输出能直接复制到催办邮件/消息/工单
